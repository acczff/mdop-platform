> 系统简称：WMS
>
> 所属平台：制造业数字化运营平台（MDOP）
>
> 文档版本：V0.4
>
> 当前状态：总体业务范围仍在分析；采购到货收货方案已确认，但尚未实现
>
> 最近更新：2026-07-20

## 0. 分析原则

- 先确认业务，再设计技术实现。
- 库存与仓储流程可以使用模拟数据完整验证。
- 生产工艺和生产管理规则必须结合具体企业调研，当前不凭空模拟。
- ERP、MES和QMS第一期只通过模拟接口或模拟事件参与，不实现其内部流程。
- 新需求在后续独立分析和排期，不直接打断当前学习迭代。
- 同时保留业务优先级和学习实施顺序，二者不混为一谈。

# 第一部分：为什么建设这个系统

## 1. 系统背景

### 1.1 企业模型

本项目以一家小批量、多品种的智能电子设备整机装配企业为业务背景，由客户订单驱动生产。

企业可能使用的主要物料包括电子元器件、PCB、线材、结构件、包装材料、半成品和整机成品。不同企业的SMT、DIP、烧录、组装、测试、老化及包装等生产工艺存在较大差异，因此本WMS不预设具体生产工艺。

### 1.2 当前业务痛点

以下痛点全部纳入WMS总体范围：

| 编号 | 业务痛点 | 业务优先级 |
|---|---|---:|
| PAIN-001 | 系统库存与实物库存不一致 | P0 |
| PAIN-002 | 生产领料后，无法判断物料仍在总仓、已到线边仓，还是已由生产系统确认消耗 | P0 |
| PAIN-003 | 同一库存可能被多个销售订单或生产工单重复占用 | P0 |
| PAIN-004 | 入库、出库依赖纸质单据或Excel，过程不规范 | P1 |
| PAIN-005 | 重复提交或多人操作可能造成重复入库、重复出库 | P0 |
| PAIN-006 | 盘点耗时长，发现差异后无法定位原因 | P1 |
| PAIN-007 | 销售订单、生产工单与库存变化无法关联 | P0 |
| PAIN-008 | 仓库异常后无法追踪操作者、操作时间和具体动作 | P1 |

### 1.3 总体建设目标

1. **库存透明**：能够查询任意物料当前所在仓库、库区、库位、批次和库存状态。
2. **账实可控**：所有库存变化都通过业务单据和库存流水完成，禁止普通用户直接修改库存余额。
3. **生产可追踪**：在WMS职责范围内，能够追踪物料从原材料仓、拣货、交付线边、退料或转入异常区域的过程；生产实际消耗只能依据MES回传结果记录。
4. **操作可靠**：避免库存重复占用、重复入库、重复出库和负库存。
5. **全程可审计**：库存变化能够追溯到来源订单、生产工单、操作人员和操作时间。

### 1.4 第一版成功指标

当前没有真实企业运行基线，第一版不编造效率提升比例，先使用可以通过自动化测试和数据对账证明的正确性指标。

| 指标 | 目标 | 验证方式 |
|---|---:|---|
| 核心库存变化生成业务单据和库存流水 | 100% | 集成测试与数据检查 |
| 库存流水可关联来源订单或生产工单 | 100% | 业务链路测试 |
| 关键操作记录操作者、时间、业务对象和结果 | 100% | 审计日志检查 |
| 同一幂等请求重复执行产生重复库存变化 | 0次 | 重复请求测试 |
| 并发库存操作产生负库存 | 0次 | 并发测试 |
| 已占用库存超过实际库存 | 0次 | 并发测试与约束检查 |
| 库存余额与流水汇总无法解释的差异 | 0项 | 自动对账 |
| WMS职责范围内无法判断物料位置或库存状态 | 0项 | 端到端流程测试 |
| 非授权用户成功完成敏感操作 | 0次 | 权限测试 |

后续接入真实企业数据后，再评估账实相符率、平均入库时间、平均出库时间、盘点耗时、拣货准确率和库存周转率等经营指标。

## 2. 系统范围与边界

### 2.1 第一期仓库范围

| 仓库 | 主要用途 |
|---|---|
| 原材料仓 | 存放电子元器件、PCB、线材、结构件和包装材料等生产物料 |
| 半成品仓 | 存放已完成部分加工或装配、等待后续处理的半成品 |
| 成品仓 | 存放检验合格、等待销售出库的整机成品 |
| 线边仓 | 存放已经从总仓发出、等待生产使用或等待退料的物料 |
| 备件耗材仓 | 存放设备维修备件、工具和低值耗材 |

### 2.2 第一期核心库区

- 收货暂存区
- 待检区
- 合格品区
- 不合格品区
- 待上架区
- 拣货暂存区
- 待发区
- 退货区
- 返工区
- 报废暂存区

### 2.3 库存表达维度

同一批物料至少需要同时表达：

```text
存放位置：仓库 / 库区 / 库位
库存状态：待检 / 合格 / 冻结 / 不合格 / 报废等
业务属性：供应商批次 / 生产批次 / 序列号 / 所有权
```

库存状态不通过无限增加“虚拟仓库”表达，仓库位置和质量状态分别建模。

### 2.4 第一期外部系统

| 外部系统 | 向WMS提供 | WMS向其反馈 | 第一期实现方式 |
|---|---|---|---|
| ERP | 采购到货、销售出库需求及来源单号 | 实际入库、实际出库及库存结果 | 模拟接口或模拟事件 |
| MES | 生产领料、退料、成品入库需求及生产工单号 | 发料、退料和成品入库结果 | 模拟接口或模拟事件 |
| QMS | 来料、过程物料或成品的质量判定结果 | 待检物料、库存状态变化结果 | 模拟接口或模拟事件 |

### 2.5 WMS负责的生产协同范围

WMS负责：

- 接收生产领料需求。
- 校验、占用和释放库存。
- 生成拣货任务并完成拣货、复核。
- 将物料从原材料仓转移到线边仓。
- 记录生产退料、余料退回和异常物料转移。
- 根据MES明确回传的结果记录生产消耗相关库存结果。

WMS不负责：

- 计算生产物料需求。
- 定义BOM和替代料规则。
- 判断物料是否投入具体工序。
- 管理工序进度、报工、损耗和完工数量。
- 判断生产工艺和损耗是否合理。

上述内容属于MES、APS或企业具体生产管理规则，必须通过企业调研后另行设计。

### 2.6 第一期暂不实现

- 具体电子制造生产工艺流程
- 自动化立体仓库
- AGV调度
- RFID
- 智能货柜
- 恒温恒湿设备联动
- 湿敏元器件开封时间管理
- 危险品专业管理
- 多工厂跨法人库存

以上能力保留为后续需求，不影响第一期核心仓储模型设计。

业务痛点全部保留，按照从基础到企业级可靠性的顺序解决：

```text

```

# 第二部分：业务如何运行

## 4. 角色与职责

### 4.1 人工角色

| 角色 | 主要职责 |
|---|---|
| 仓库主管 | 审核业务单据、分配仓储任务、审批库存调整、处理重大异常 |
| 收货员 | 接收供应商来料，核对并登记实收物料、数量和批次信息 |
| 上架员 | 根据上架任务将合格物料放入指定库位，并确认实际上架结果 |
| 拣货员 | 按销售出库或生产领料任务执行拣货，并登记实拣数量和批次 |
| 复核员 | 复核已拣物料、批次、数量和目标业务单据 |
| 盘点员 | 执行盘点任务，录入实物盘点数量和异常说明 |
| 质量检验员 | 在QMS中完成检验，并由QMS向WMS提供质量判定结果 |
| 生产领料员 | 接收交付到线边仓的物料，配合处理余料退回和生产退料 |
| 库存管理员 | 维护仓库层级，处理调拨、冻结、解冻、差异复核、库存调整、对账和异常调查 |
| 系统管理员 | 维护用户、角色和权限，不默认拥有仓储业务操作权限 |

### 4.2 外部系统参与者

| 参与者 | 参与方式 |
|---|---|
| ERP | 提供采购到货和销售出库需求，接收实际出入库结果 |
| MES | 提供生产领料、退料和成品入库需求，接收仓储执行结果 |
| QMS | 提供质量判定结果，驱动待检库存的质量状态变化 |

### 4.3 职责分离规则

| 规则编号 | 规则 |
|---|---|
| RULE-AUTH-001 | 单据创建人不能审核自己创建的单据 |
| RULE-AUTH-002 | 拣货员不能复核自己完成的拣货任务 |
| RULE-AUTH-003 | 盘点员不能审批自己录入的盘点差异 |
| RULE-AUTH-004 | 库存管理员可以发起库存调整，但必须由仓库主管审批 |
| RULE-AUTH-005 | 普通仓库人员不能直接修改库存余额 |
| RULE-AUTH-006 | 系统管理员管理账号和权限，但不默认拥有仓储业务操作权限 |
| RULE-AUTH-007 | 质量状态依据QMS判定结果变更，仓库人员不能自行将不合格品改为合格品 |
| RULE-AUTH-008 | 已完成的出入库业务不能删除或直接修改数量，只能通过退货、冲正或调整流程处理 |

### 5.4 异常类场景

| 场景编号 | 异常场景 | 预期处理方向 |
|---|---|---|
| SCN-EX-001 | 入库数量或批次不符 | 记录差异，阻止无依据的强行完成，进入确认或异常处理 |
| SCN-EX-002 | 拣货数量不足 | 记录短拣，重新分配库存或等待业务决策 |
| SCN-EX-003 | 复核不通过 | 退回重新拣货或进入差异处理，不允许直接完成出库 |

| SCN-EX-005 | 并发占用库存 | 通过数据库并发控制保证占用总量不超过实际库存 |
| SCN-EX-006 | 外部系统重复推送、超时或不可用 | 支持幂等接收、重试、补偿、告警和人工处理 |

场景先完整保留，再按照学习实施顺序逐层实现；后续新增场景需要重新分析边界和优先级。

## 6. SCN-IN-001 采购到货收货

### 6.1 业务目的与边界

采购到货收货用于依据ERP中已审核的采购业务接收供应商来料，登记实际到货数量和批次，使实物正式进入企业库存控制范围，并为后续QMS来料检验提供依据。

正常起点：

```text
ERP采购订单已审核
→ ERP生成采购到货通知
→ WMS接收到货通知
→ 供应商送货
→ 收货员依据到货通知收货
```

WMS不创建或审核采购订单，只保存ERP采购订单号、到货通知单号、供应商、预计物料、预计数量和计划到货时间等来源信息。

无来源到货不能直接进入普通收货流程，必须进入异常收货，由仓库主管确认，并在后续补充ERP来源依据。

### 6.2 前置条件

- ERP采购订单已经审核。
- ERP已经生成采购到货通知。
- 供应商、物料、计量单位等主数据有效。
- WMS能够根据外部来源标识判断到货通知是否已经接收。
- 收货暂存区或待检区处于可用状态。
- 当前用户具备采购收货权限。

### 6.3 正常流程

1. ERP将已审核的采购到货通知推送给WMS。
2. WMS校验外部来源标识，防止重复创建到货任务。
3. WMS生成待收货任务，初始状态为`PENDING_RECEIPT`。
4. 供应商送货后，收货员查找对应到货通知。
5. 收货员核对供应商、物料、通知数量、计量单位和外包装。
6. 收货员登记本次实收数量、供应商批次及收货暂存位置。
7. 存在少货、错料、超量、批次或包装异常时，记录收货差异。
8. 收货员提交本次收货结果。
9. WMS生成待检库存和采购收货库存流水，该库存不可用于领料或销售出库。
10. WMS向QMS发出来料检验需求，并向ERP反馈实际收货结果。

### 6.4 数量、物料与批次规则

| 规则编号 | 业务规则 |
|---|---|
| RULE-REC-001 | 允许一张到货通知分多次收货，累计实收数量不得超过通知数量 |
| RULE-REC-002 | 允许少收，剩余数量继续保持待收；ERP关闭剩余数量后，WMS同步关闭任务 |
| RULE-REC-003 | 默认禁止超收，ERP调整到货通知数量后才能继续正常收货 |
| RULE-REC-004 | 到货物料不在通知明细中时进入异常处理，不能直接加入正常库存 |
| RULE-REC-005 | 同一种物料存在多个供应商批次时，必须拆分为多条收货批次明细 |
| RULE-REC-006 | 数量、物料、批次或包装不符必须记录差异类型、实际情况、操作人员和处理结果 |
| RULE-REC-007 | 收货完成后物料进入待检状态，不能直接成为可用库存 |

### 6.5 必要业务数据

- ERP采购订单号
- ERP到货通知单号
- WMS收货业务单号
- 供应商
- 物料编码和名称
- 通知数量、累计已收数量、本次实收数量
- 计量单位
- 供应商批次号
- Date Code（适用时）
- 生产日期和有效期（适用时）
- 包装数量和包装状态
- 收货暂存库位
- 收货差异类型和说明
- 收货员和收货时间

物料管理粒度由物料主数据策略决定：

- 电子元器件、PCB和线材等原材料优先按批次管理。
- 整机成品可以按序列号管理。
- 普通低值耗材可以只按数量管理。
- 是否需要批次、Date Code、有效期或序列号，由物料配置决定。

### 6.6 收货任务状态机

| 状态 | 业务含义 |
|---|---|
| PENDING_RECEIPT | 已接收到货通知，尚未开始收货 |
| RECEIVING | 收货员正在登记本次实际到货 |
| PARTIALLY_RECEIVED | 已完成部分收货，仍有剩余待到货 |
| RECEIVED | 到货通知数量已经全部收货 |
| CLOSED | ERP确认剩余数量不再到货，任务正常关闭 |
| CANCELLED | 尚未发生有效收货前，到货通知被ERP取消 |

主要迁移：

```text
PENDING_RECEIPT
→ RECEIVING
→ PARTIALLY_RECEIVED
→ RECEIVING
→ RECEIVED

PARTIALLY_RECEIVED → CLOSED
PENDING_RECEIPT → CANCELLED
```

收货差异单独记录，不作为整张任务的主状态。一张到货任务可以同时包含正常收货明细和异常差异明细。

已经产生有效收货库存后，不能直接取消整张任务，只能通过退货或冲正流程处理。

### 6.7 收货确认与库存规则

| 规则编号 | 业务规则 |
|---|---|
| RULE-REC-008 | 收货员提交前可以修改本次收货草稿 |
| RULE-REC-009 | 提交收货时，收货明细、累计数量、待检库存、库存流水和操作日志必须作为一个整体成功或失败 |
| RULE-REC-010 | 待检库存属于企业现有库存，但可用数量为0 |
| RULE-REC-011 | 待检库存禁止销售出库、生产领料和普通调拨 |
| RULE-REC-012 | 待检库存只能处于收货暂存区或待检区 |
| RULE-REC-013 | 收货提交后不能直接修改物料、批次和数量 |
| RULE-REC-014 | 提交后发现错误，必须通过收货冲正生成反向流水，不删除原记录 |
| RULE-REC-015 | 已被QMS检验或进入后续上架流程的收货记录不能直接冲正，必须进入退货或对应异常流程 |

### 6.8 库存结果

收货确认后的数量关系：

```text
现有库存增加
待检库存增加
可用库存不增加
```

“收货完成”只表示货物已经进入企业控制范围，不表示质量合格，也不表示物料可以被生产或销售使用。

### 6.11 业务验收场景

#### AC-REC-001 正常收货

```text
给定ERP通知物料A到货100件
当收货员登记同一批次实收100件并提交
那么任务状态为RECEIVED
并且待检库存增加100
并且可用库存增加0
并且生成一条采购收货流水
```

#### AC-REC-002 分批收货

```text
给定ERP通知到货100件
当第一次收货60件
那么任务状态为PARTIALLY_RECEIVED
并且剩余待收40件

当第二次收货40件
那么任务状态为RECEIVED
并且累计待检库存为100件
```

#### AC-REC-003 禁止超收

```text
给定通知剩余数量为40件
当收货员提交50件
那么系统拒绝提交
并且库存、流水和累计收货数量都不变化
```

#### AC-REC-004 错误物料

```text
给定到货通知不包含物料B
当收货员尝试将物料B加入正常收货
那么系统拒绝
并且允许登记异常差异
```

#### AC-REC-005 批次校验

```text
给定物料A配置为批次管理
当收货员未填写供应商批次
那么系统拒绝提交
```

#### AC-REC-006 重复提交

```text
给定某次收货请求已经成功
当客户端使用相同幂等键再次提交
那么返回首次处理结果
并且库存和流水不能重复增加
```

#### AC-REC-007 权限校验

```text
给定当前用户没有采购收货权限
当其提交收货结果
那么系统返回403
并且所有业务数据不发生变化
```

#### AC-REC-008 事务回滚

```text
给定收货提交过程中库存流水写入失败
那么收货明细、累计数量、库存余额和操作结果全部回滚
并且不能出现部分成功
```

#### AC-REC-009 收货冲正

```text
给定一笔尚未质检的收货已经完成
当冲正申请审批通过
那么生成反向库存流水
并且待检库存恢复到收货前数量
并且原收货记录继续保留
```

#### AC-REC-010 禁止错误冲正

```text
给定收货物料已经由QMS完成检验
当用户申请直接冲正
那么系统拒绝
并且提示进入退货或异常处理流程
```

## 7. 采购收货领域模型

### 7.1 核心业务对象

| 业务对象 | 类型 | 主要职责 |
|---|---|---|
| 到货通知 `ArrivalNotice` | 聚合根 | 保存ERP到货通知快照、累计收货数量和任务状态，控制分批收货、关闭和取消 |
| 到货通知明细 `ArrivalNoticeItem` | 实体 | 保存物料、通知数量、累计已收数量和剩余待收数量 |
| 收货记录 `Receipt` | 聚合根 | 表示一次实际收货提交，保存收货单号、收货人、时间和处理结果 |
| 收货明细 `ReceiptItem` | 实体 | 保存本次实收物料、数量、批次、Date Code和暂存库位 |
| 收货差异 `ReceivingDiscrepancy` | 实体或独立聚合 | 记录少货、超收、错料、批次、包装等异常及处理结果 |
| 库存余额 `InventoryBalance` | 聚合根 | 保存某物料在特定仓库、库位、批次和质量状态下的当前数量 |
| 库存流水 `InventoryTransaction` | 不可变业务记录 | 记录库存变化前数量、变化数量、变化后数量和来源收货单 |
| 收货冲正 `ReceiptReversal` | 聚合根 | 管理冲正申请、审批和反向库存处理 |
| 来料检验请求 `IncomingInspectionRequested` | 领域事件 | 告知QMS某批物料已收货，需要进行质量检验 |

对象关系：

```text
ArrivalNotice
├─ ArrivalNoticeItem
└─ 允许产生多次 Receipt

Receipt
├─ ReceiptItem
├─ ReceivingDiscrepancy
├─ 更新 InventoryBalance
├─ 生成 InventoryTransaction
└─ 发布 IncomingInspectionRequested

Receipt
└─ 发生错误时产生 ReceiptReversal
```

边界说明：

- ERP采购订单不是WMS实体，WMS只保存采购订单号和必要快照。
- 供应商、物料和库位属于主数据，收货流程引用其ID和必要快照。
- 一次收货提交对应一张独立`Receipt`，分批到货会产生多张收货记录。
- 到货通知负责累计数量，收货记录负责保存每次实际发生的业务事实。
- 库存流水创建后不可修改或删除。

### 7.2 领域不变量

| 编号 | 不变量 |
|---|---|
| INV-REC-001 | 到货通知累计已收数量不能小于0 |
| INV-REC-002 | 正常收货累计数量不能超过到货通知数量 |
| INV-REC-003 | 每次正常收货必须关联有效到货通知及明细；异常收货必须关联已批准的异常依据 |
| INV-REC-004 | 一张收货单至少包含一条有效收货明细 |
| INV-REC-005 | 收货数量必须大于0 |
| INV-REC-006 | 批次管理物料必须填写批次；序列号管理物料的序列号数量必须等于实收数量 |
| INV-REC-007 | 同一个序列号不能重复入库 |
| INV-REC-008 | 收货库存的物料、批次、库位和质量状态必须与库存流水一致 |
| INV-REC-009 | 收货提交时，到货累计数量、库存余额、库存流水和收货记录必须全部成功或全部失败 |
| INV-REC-010 | 库存流水创建后不可修改、不可删除 |
| INV-REC-011 | 已提交收货单不能修改物料和数量 |
| INV-REC-012 | 冲正数量不能超过原收货尚未被后续业务使用的可冲正数量 |
| INV-REC-013 | 同一个幂等键只能对应一个确定的收货处理结果 |
| INV-REC-014 | ERP到货通知外部来源编号在WMS内必须唯一 |
| INV-REC-015 | 待检库存的可用数量必须为0 |

这些不变量后续分别通过领域方法、数据库约束、唯一索引、事务和自动化测试保护，不全部堆积在Controller中。

## 8. 采购收货数据模型

### 8.1 核心业务表

| 表名 | 业务用途 |
|---|---|
| `wms_arrival_notice` | 到货通知主表，保存ERP来源、供应商和任务状态 |
| `wms_arrival_notice_item` | 到货通知明细，保存通知数量、累计已收数量 |
| `wms_receipt` | 每一次实际收货记录 |
| `wms_receipt_item` | 实收物料、数量、批次、Date Code和暂存库位 |
| `wms_receipt_serial` | 序列号管理物料的逐件序列号 |
| `wms_receiving_discrepancy` | 数量、物料、批次和包装差异 |
| `wms_inventory_balance` | 按物料、库位、批次、质量状态汇总的库存余额 |
| `wms_inventory_transaction` | 不可变库存流水 |
| `wms_receipt_reversal` | 收货冲正申请和审批记录 |

### 8.2 关键唯一约束

```text
wms_arrival_notice：
source_system + external_notice_no 唯一

wms_receipt：
receipt_no 唯一
idempotency_key 唯一

wms_receipt_serial：
serial_no 唯一

wms_inventory_balance：
warehouse_id
+ location_id
+ material_id
+ batch_no
+ quality_status
+ owner_type
+ owner_id
唯一
```

### 8.3 数据设计原则

- 到货通知与实际收货分开，因为一张通知允许多次收货。
- 收货主表与明细分开，因为一次收货可以包含多种物料和批次。
- 序列号单独存表，不把大量序列号塞入字符串或JSON字段。
- 库存余额用于快速查询，库存流水用于追溯和对账。
- 冲正保留原流水并生成反向流水，不覆盖历史事实。
- 第一版正式增加Inbox、Outbox及消息投递记录，用于接收去重、可靠发布、重试、死信、重放和对账。
- 字段类型、完整索引和SQL迁移在查询场景、容量假设和并发方案确认后设计。

## 10. 模块边界

```text
wms
├─ inbound
│  ├─ 到货通知
│  ├─ 收货
│  ├─ 收货差异
│  └─ 收货冲正
├─ inventory
│  ├─ 库存余额
│  ├─ 库存流水
│  └─ 库存查询
└─ integration
   ├─ ERP事件适配
   ├─ ERP收货结果发布
   └─ QMS检验请求发布
```

依赖规则：

- `inbound`通过`inventory`公开的应用接口完成库存入账，不能直接访问其Repository。
- `integration`负责将ERP事件转换为WMS内部命令，并将WMS领域事件转换为外部事件。
- ERP和QMS的消息格式不能渗透到领域对象。
- `inventory`不依赖采购订单或QMS的具体数据结构。
- Controller只负责协议转换、参数校验和调用应用用例。
- 业务规则放在领域对象或领域服务，事务编排放在应用服务。
- JPA、RabbitMQ和外部适配器实现放在基础设施层。
- 当前保持模块化单体，不因采用RabbitMQ而拆成伪微服务。

## 11. 正式跨系统集成方案

### 11.1 ERP向WMS发布事件

### 11.3 正式可靠性能力

第一版直接包含：

- 消息唯一ID
- Inbox接收记录和消费幂等
- 事务Outbox
- 发布失败重试

## 12. 提交收货事务边界

提交收货使用一个本地数据库事务，事务内依次完成：

```text
1. 校验幂等键
2. 加载并校验到货通知状态
3. 校验物料、数量、批次和序列号
4. 保存收货单及明细
5. 更新到货通知累计已收数量
6. 增加待检库存余额
7. 写入不可变库存流水
8. 保存收货差异
9. 写入操作审计
10. 写入正式Outbox事件
```

任意一步失败时，收货单、累计数量、库存余额、库存流水、审计记录和Outbox事件全部回滚。

事务外执行：

- RabbitMQ实际消息投递
- ERP和QMS实际消费
- 通知发送
- 文件处理
- 其他远程调用

远程调用不进入本地数据库事务。数据库事务成功但MQ暂时不可用时，由Outbox发布任务持续重试；数据库事务失败时不会留下待发送事件。

## 13. 并发控制方案

### 13.1 防止累计超收

到货通知明细采用数据库条件更新：

```sql
UPDATE wms_arrival_notice_item
SET received_qty = received_qty + :qty
WHERE id = :id
  AND received_qty + :qty <= notice_qty;
```

受影响行数为0表示数量已被其他事务改变或本次收货会造成超收，业务必须拒绝并重新读取最新数据。

### 13.2 防止库存余额丢失更新

- 根据仓库、库位、物料、批次、质量状态和所有权定位唯一库存余额。
- 在事务内使用数据库行锁锁定余额行后更新数量。
- 多条库存维度按固定顺序加锁，降低死锁概率。
- 余额维度建立唯一索引，防止并发创建重复余额。
- 首次创建余额行时必须处理并发插入的唯一键冲突，并重新读取已有余额。

### 13.3 状态与重试

- 收货任务和冲正申请使用版本号进行乐观锁控制，防止旧页面覆盖新状态。
- 只对明确可安全重试的数据库死锁进行有限重试。
- 重试必须沿用原幂等键，不能产生新的业务结果。
- Redis分布式锁不作为库存正确性的最终保障；数据库事务、约束和锁才是最终数据防线。

## 14. 消息可靠性与幂等

### 14.1 投递语义

RabbitMQ采用至少一次投递。系统不承诺底层消息只出现一次，而是通过Inbox、业务唯一约束和幂等键保证业务最终只生效一次。

### 14.2 消息公共字段

每条跨系统消息必须包含：

```text
messageId
eventType
eventVersion
sourceSystem
occurredAt
traceId
aggregateType
aggregateId
payload
```

### 14.3 Inbox规则

- WMS接收ERP事件时先写入Inbox，`messageId`建立唯一约束。
- 相同`messageId`再次到达时不重复执行业务，返回或记录已有处理结果。
- 即使`messageId`不同，ERP到货通知仍通过来源系统和外部通知号保证业务唯一。
- 消费者业务处理成功后才确认RabbitMQ消息。
- 消费处理失败时回滚业务事务，由RabbitMQ重新投递。

### 14.4 Outbox规则

- Outbox与业务数据在同一个本地事务中写入。
- Outbox状态流转：

```text
PENDING → PUBLISHING → PUBLISHED
                  └→ FAILED → DEAD
```

- 发布失败采用有限次数的指数退避重试。
- 超过重试次数进入死信，由运维页面查询、修复和人工重放。
- 人工重放沿用原`messageId`或原业务幂等键，不能重复改变库存。

### 14.5 对账与补偿

定时对账至少检查：

- 业务成功但Outbox长期未发布。
- Outbox已发布但下游长期没有处理结果。
- Inbox处理失败或长时间处于处理中。
- ERP到货数据、WMS收货结果与反馈状态不一致。
- QMS检验请求没有被正常接收。

异常必须支持告警、自动重试、人工处理和处理结果留痕。

## 15. 安全与审计

- 使用Spring Security建立统一认证授权，不引入可伪造的自定义Token方案。
- 采购收货采用细粒度权限编码：
  - `wms:arrival:read`
  - `wms:receipt:create`
  - `wms:receipt:submit`
  - `wms:receipt:reverse:apply`
  - `wms:receipt:reverse:approve`
  - `wms:discrepancy:handle`
- 用户除接口权限外，还必须具备目标仓库的数据权限。
- 收货员只能操作被授权仓库的到货任务。
- 冲正申请人与审批人不能相同。
- 敏感操作记录操作前后状态、业务单号、用户、时间、IP、TraceId和结果。
- 库存流水、审计日志和消息处理记录不能由普通业务用户删除。
- ERP、WMS、QMS使用独立RabbitMQ账号及最小队列权限。
- 消息来源和事件版本必须校验，未知来源或不支持版本不进入正常业务。
- 密码、RabbitMQ凭据和JWT密钥由环境变量或密钥管理提供，不写入代码仓库。
- 死信重放需要独立权限，并记录重放原因和处理结果。
- 查询序列号、供应商批次等敏感追溯信息需要授权。

## 16. 可观测性

### 16.1 链路标识

系统必须能够通过以下任意标识追踪完整链路：

- ERP到货通知号
- WMS收货单号
- 消息ID
- TraceId
- 物料批次或序列号
- 库存流水号

### 16.2 日志

- 使用结构化JSON日志。
- 自动记录TraceId、用户ID、消息ID和业务单号。
- 记录状态迁移、库存变化、消息重试和异常原因。
- 禁止输出密码、JWT、RabbitMQ凭据等敏感信息。

### 16.3 指标

- 待收货任务和长期部分收货任务数量
- 收货提交成功率和响应时间
- 收货差异数量
- Outbox待发送、失败和死信数量
- Inbox处理失败和积压数量
- RabbitMQ队列积压量
- 库存对账差异数量
- 数据库事务失败和死锁次数

### 16.4 告警

- Outbox事件长时间未发布
- RabbitMQ队列持续积压
- 消息多次重试进入死信
- Inbox长时间处于处理中
- 库存余额与流水出现差异
- 到货通知长期部分收货
- 短时间内出现大量收货冲正
- 接口错误率或处理时间超过阈值

### 16.5 运维查询

- 根据TraceId查询完整请求链路。
- 根据消息ID查询生产、投递、消费和重试记录。
- 根据业务单号查询收货、库存流水、QMS请求和ERP反馈。
- 授权用户可以执行人工重试或死信重放。

## 17. REST接口契约

统一版本前缀：`/api/v1`。

### 17.1 到货任务

```text
GET /api/v1/wms/arrival-notices
GET /api/v1/wms/arrival-notices/{id}
GET /api/v1/wms/arrival-notices/{id}/receipts
```

### 17.2 收货操作

```text
POST /api/v1/wms/arrival-notices/{id}/receipts

PUT  /api/v1/wms/receipts/{id}
POST /api/v1/wms/receipts/{id}/submit
```

创建接口生成收货草稿；只有`submit`才真正增加待检库存并写入库存流水。

### 17.3 收货差异与冲正

```text
POST /api/v1/wms/receipts/{id}/discrepancies
POST /api/v1/wms/receipts/{id}/reversals
POST /api/v1/wms/receipt-reversals/{id}/approve

```

### 17.4 库存查询

```text
GET /api/v1/wms/inventory-balances
GET /api/v1/wms/inventory-transactions
```

### 17.5 消息运维

```text
GET  /api/v1/ops/inbox-messages
GET  /api/v1/ops/outbox-events
GET  /api/v1/ops/dead-letters
POST /api/v1/ops/messages/{id}/replay
```

### 17.6 REST设计规则

- 查询使用`GET`，创建草稿使用`POST`，修改草稿使用`PUT`。
- 提交、审批、驳回、重放使用明确的`POST`业务命令。
- 已提交收货单不提供普通修改和删除接口。
- 跨系统正式集成使用RabbitMQ，不暴露ERP推送REST接口。
- 写接口携带幂等键或版本号。
- 统一返回稳定业务错误码，404不能表示登录失效。

## 18. 正式事件契约

### 18.1 统一事件信封

```json
{
  "messageId": "UUID",
  "eventType": "PurchaseArrivalNoticeCreated",
  "eventVersion": 1,
  "sourceSystem": "ERP",
  "occurredAt": "2026-07-11T10:00:00+08:00",
  "traceId": "TRACE-ID",
  "aggregateType": "PurchaseArrivalNotice",
  "aggregateId": "ERP-NOTICE-001",
  "payload": {}
}
```

### 18.2 ERP到WMS

| 事件 | 核心数据 |
|---|---|
| `PurchaseArrivalNoticeCreated` | 到货通知号、采购订单号、供应商快照、目标仓库、计划到货时间、物料明细、数量、单位、ERP业务版本 |
| `PurchaseArrivalNoticeChanged` | 外部通知号、ERP业务版本、允许变更字段、变更原因 |
| `PurchaseArrivalNoticeCancelled` | 外部通知号、ERP业务版本、取消原因 |
| `PurchaseArrivalNoticeClosed` | 外部通知号、ERP业务版本、关闭剩余数量、关闭原因 |

### 18.3 WMS到ERP

| 事件 | 核心数据 |
|---|---|
| `ReceiptCompleted` | 收货单号、到货通知号、采购订单号、仓库、收货时间和人员、实收物料、数量、单位、批次及差异摘要 |
| `ReceiptReversed` | 原收货单号、冲正单号、冲正数量、原因、审批人和时间 |

### 18.4 WMS到QMS

`IncomingInspectionRequested`包含检验请求号、收货单号、供应商、仓库、待检位置、物料、数量、单位、批次、Date Code、生产日期、有效期和差异摘要。

### 18.5 事件版本规则

- 消费者明确声明支持的`eventVersion`。
- 新增非必填字段可以保持兼容。
- 删除字段、修改含义或修改类型必须升级事件版本。
- 不支持的版本进入隔离队列并告警，不能直接丢弃。
- 消息只携带必要业务事实，不传输JPA实体。

## 19. HTTP状态与业务错误码

### 19.1 HTTP状态语义

| HTTP状态 | 含义 |
|---:|---|
| 400 | 请求格式或参数校验失败 |
| 401 | 未登录或认证失效 |
| 403 | 已登录但无操作或数据权限 |
| 404 | 业务资源不存在 |
| 409 | 状态、并发或幂等冲突 |
| 422 | 请求格式正确但违反业务规则 |
| 503 | RabbitMQ等基础设施暂时不可用 |

### 19.2 当前错误码注册表

| 错误码 | 含义 |
|---|---|
| `WMS-REC-001` | 到货通知不存在 |
| `WMS-REC-002` | 当前状态不允许执行该操作 |
| `WMS-REC-003` | 本次收货数量超过剩余可收数量 |
| `WMS-REC-004` | 物料不属于当前到货通知 |
| `WMS-REC-005` | 批次管理物料缺少批次 |
| `WMS-REC-006` | 序列号缺失或重复 |
| `WMS-REC-007` | 收货单已经提交，不能修改 |
| `WMS-REC-008` | 幂等键已用于不同请求内容 |
| `WMS-REC-009` | 无目标仓库的数据权限 |
| `WMS-REC-010` | 当前收货记录不允许冲正 |
| `WMS-REC-011` | 冲正申请人与审批人不能相同 |
| `WMS-REC-012` | 数据已被其他事务修改，请刷新后重试 |
| `WMS-INV-001` | 库存余额不存在 |
| `WMS-INV-002` | 库存余额与流水对账不一致 |
| `WMS-MSG-001` | 消息来源不受信任 |
| `WMS-MSG-002` | 事件类型不受支持 |
| `WMS-MSG-003` | 事件版本不受支持 |
| `WMS-MSG-004` | 消息处理失败并进入重试 |
| `WMS-MSG-005` | 消息已进入死信 |
| `WMS-MSG-006` | 消息不满足人工重放条件 |

### 19.3 动态维护规则

- 错误码注册表可以随着新场景持续增加。
- 已发布错误码不得改变含义，不得删除后分配给其他错误。
- 废弃错误码标记为Deprecated并保留历史说明。
- 新错误码必须说明所属模块、HTTP状态、触发条件、用户提示和是否可重试。
- 错误码变化必须同步更新前端映射、接口文档、自动化测试、监控规则和变更记录。
- 用户提示文案可以优化，稳定错误码和语义不能随意改变。
- 相同幂等请求返回原成功结果；相同幂等键对应不同请求内容才返回冲突。
- 日志保存技术详情，前端不暴露堆栈、SQL或内部基础设施信息。
