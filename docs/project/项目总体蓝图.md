# 制造业数字化运营平台项目总体蓝图

> 文档状态：V1.0
>
> 项目代号：MDOP（Manufacturing Digital Operations Platform）
>
> 核心目标：建设一个覆盖制造企业主要经营与生产流程的企业级信息化项目集合体，并将其作为长期学习、技术实践、项目管理和职业能力证明的平台。

## 1. 项目愿景

本项目模拟一家离散制造企业，从客户需求进入，到销售、计划、采购、生产、质量、仓储、配送、设备管理和经营分析的完整数字化过程。

项目不以“功能数量最多”为目标，而以以下结果为准：

- 业务流程能够形成端到端闭环。
- 关键数据正确、可追溯、可对账。
- 权限边界清晰，敏感操作可审计。
- 异常能够发现、定位、恢复和补偿。
- 代码具备清晰边界，可以测试、维护和持续演进。
- 系统具备标准化构建、部署、监控和回滚能力。
- 项目中的关键设计能够由项目负责人独立解释和取舍。

## 2. 项目定位

### 2.1 项目是什么

这是一个“统一平台 + 多个业务系统”的制造业企业应用集合体：

```text
制造业数字化运营平台
├─ 平台基础能力
├─ IAM 统一身份与权限
├─ MDM 主数据管理
├─ CRM 客户与销售
├─ ERP 订单、采购与结算
├─ APS 生产计划与排程
├─ MES 生产执行
├─ QMS 质量管理
├─ WMS 仓储管理
├─ TMS 运输配送
├─ EAM 设备资产与维修
├─ IoT 设备接入
└─ BI 数据分析
```

### 2.2 项目不是什么

- 不复制SAP、用友或大型MES的全部功能。
- 不实现完整财务总账、税务和薪资系统。
- 不在业务边界尚未稳定时拆成大量微服务。
- 不为展示技术名词而引入没有实际价值的中间件。
- 不一次性开发所有系统，而是沿核心业务链路逐步扩展。

## 3. 总体业务主线

第一条核心链路：

```text
客户建档
→ 销售订单
→ 生产需求
→ 生产计划
→ 生产工单
→ 工序与报工
→ 完工质检
→ 成品入库
→ 销售出库
→ 配送签收
→ 客户对账
```

第二条物料保障链路：

```text
生产需求
→ BOM展开
→ 物料需求
→ 采购申请
→ 采购订单
→ 供应商到货
→ 来料检验
→ 原料入库
→ 生产领料
```

第三条设备保障链路：

```text
设备建档
→ 点检与保养
→ 状态采集
→ 故障告警
→ 故障报修
→ 维修派工
→ 维修执行
→ 验收关闭
```

### 4.12 IoT 设备接入

- 设备注册与身份认证

### 4.13 BI 数据分析

- 订单交付率

- 库存周转

## 5. 架构原则

### 5.1 业务系统不等于微服务

业务上划分为多个系统，技术上先采用模块化单体：

```text
一个代码仓库
一个后端部署单元
一个数据库实例
多个边界明确的业务模块
```

模块必须遵守：

- 业务模块拥有自己的实体、应用服务和Repository。
- 其他模块不能直接访问该模块的Repository。
- 跨模块通过公开应用接口或领域事件协作。
- 公共模块只能放通用能力，不能成为业务代码垃圾桶。
- 数据库表按业务域归属，Flyway只有一个权威执行入口。

### 5.2 微服务演进条件

只有满足下列条件之一才考虑拆分：

- 模块需要独立部署或高频发布。
- 模块需要独立扩容或使用不同技术模型。
- 团队形成独立负责边界。
- 数据隔离、安全或合规要求明确。
- 单体已经产生经过测量的性能或维护瓶颈。

优先候选包括IAM、IoT、文件服务和通知服务。

### 5.3 数据原则

- 业务单据状态不能被任意修改。
- 关键数量、金额和库存变化必须可追溯。
- 跨系统只传递必要数据和业务事实。
- 核心写操作必须考虑事务、并发和幂等。
- 异步一致性必须具备重试、补偿和对账能力。
- 删除策略、时间、精度、编码和审计字段统一规范。

### 5.4 安全原则

- 使用Spring Security建立统一认证授权。
- Token必须具备可靠签名、有效期和失效机制。
- 后端执行接口权限和数据权限校验。
- 密码、密钥和生产配置不得硬编码。
- 关键操作必须记录操作者、对象、时间和结果。

## 7. 建设路线

### 阶段0：新项目工程基线

目标：在独立仓库中从零建立可重复构建、测试和演进的企业级工程骨架。

- 创建独立Git仓库，旧项目只作为业务和技术参考，不复制旧代码与旧数据。
- 建立后端模块化单体骨架、Maven Wrapper和模块依赖规则。
- 建立前端pnpm工作区、管理端应用和共享包结构。
- 建立开发、测试、生产配置框架。
- 建立Flyway空库迁移、Testcontainers测试基座和质量门禁。
- 建立MySQL、RabbitMQ、Redis本地基础设施编排。
- 建立统一启动、测试、构建和部署方式。

完成标准：新环境能够根据README启动；前后端基础工程可构建、可测试；数据库能从空库自动初始化；旧项目未进入新仓库；模块边界和后续扩展方式明确。

### 阶段1：平台底座、IAM与MDM

- Spring Security认证授权
- 用户、组织、角色和数据权限
- 动态菜单和前端路由
- 操作审计
- 客户、供应商、产品、物料
- 工厂、车间、仓库、库区和库位
- 工序、设备、BOM基础模型

完成标准：形成可信的安全底座和可被业务引用的主数据。

### 阶段2：订单到生产

- 客户销售订单
- 订单审核
- 生产需求
- 简单生产计划
- 生产工单
- 工序任务
- 报工和审核

完成标准：销售订单可以追踪到工单及实际生产进度。

### 阶段3：质量、入库与出库

- 完工质检
- 不合格处理和复检
- 成品入库
- 库存余额和流水
- 库存占用
- 拣货、复核、出库
- 并发安全、幂等和库存对账

完成标准：打通“销售订单→生产→质检→入库→出库”第一条端到端链路，并通过并发和重复请求测试。

### 阶段4：采购与原料保障

- BOM物料需求
- 采购申请和采购订单
- 供应商到货
- 来料检验
- 原材料入库
- 生产领料和退料

完成标准：生产工单能够追溯其原料来源与消耗。

### 阶段5：配送、签收与对账

- 配送任务
- 发运和运输节点
- 异常配送
- 客户签收
- 客户对账和回款状态

完成标准：打通订单到签收和对账的经营闭环。

### 阶段6：设备、维修与IoT

- 设备台账
- 点检和保养
- 故障报修与维修
- MQTT/TCP设备接入
- 状态采集和告警
- 指令下发、回执、超时、重试和审计

完成标准：业务系统能够根据设备事件创建或驱动维修流程，远程指令全链路可追踪。

### 阶段7：可靠性、分析与架构演进

- Outbox可靠事件
- MQ消费幂等、重试和死信
- 定时补偿和业务对账
- 结构化日志和链路追踪
- Prometheus与Grafana监控
- 性能测试和故障演练
- BI指标与经营看板
- 根据真实瓶颈决定是否拆分服务

完成标准：系统能够观测、恢复、对账、发布和回滚。

## 8. 企业级质量门槛

每个业务功能必须至少具备：

- 明确的业务背景和使用角色。
- 正常流程、异常流程和状态迁移。
- 接口权限和数据权限。
- 数据库约束、事务边界和并发策略。
- 幂等要求和重复提交处理。
- 操作日志和关键业务流水。
- 单元测试与集成测试。
- 可验收的前端操作闭环。
- 对应接口、业务和故障排查文档。

核心链路额外要求：

- 支持端到端自动化测试。
- 支持失败回滚、补偿或人工处理。
- 支持通过业务单号追踪完整链路。
- 支持数据对账和差异定位。

## 9. 测试与交付体系

### 9.1 测试分层

- 单元测试：状态机、计算、权限和业务规则。
- 集成测试：Controller、Service、Repository及事务。
- 基础设施测试：使用Testcontainers验证MySQL、Redis和RabbitMQ。
- API测试：正常、异常、越权、重复及并发场景。
- 端到端测试：覆盖订单到出库等核心业务链路。
- 性能测试：库存扣减、报工、设备上报等重点接口。

### 9.2 交付流水线

```text
提交代码
→ 静态检查
→ 自动编译
→ 自动测试
→ 构建镜像
→ 部署测试环境
→ 数据库迁移
→ 冒烟测试
→ 人工确认
→ 生产发布或回滚
```

每个需求必须记录：

- 业务背景
- 用户角色
- 范围与非范围
- 前置条件
- 正常和异常流程
- 状态机
- 数据变化
- 权限要求
- 接口设计
- 验收标准
- 测试场景
- 发布与回滚风险

每次迭代控制在一到两周，只设一个可验收的核心成果。

每次重要技术决策使用ADR记录：

- 决策背景
- 候选方案
- 最终选择
- 选择理由
- 代价和风险
- 未来重新评估条件

## 11. 学习驱动机制

本项目同时是学习系统。学习顺序遵循：

```text
选择业务任务
→ 识别知识缺口
→ 学习必要原理
→ 完成实现
→ 编写测试验证
→ 形成文档和复盘
→ 用自己的语言讲解
```

每个迭代必须产生五类成果：

1. 可运行代码。
2. 自动化测试。
3. 业务或技术文档。
4. 一次演示或验收记录。
5. 一篇个人复盘。

禁止用观看视频时长、阅读页数或生成笔记数量代替项目成果。

## 14. 蓝图维护规则

- 本文档是项目最高层级的范围和方向依据。
- 新系统、新中间件和重大架构调整必须先更新蓝图或新增ADR。
- 详细需求、设计和学习笔记以后按实际迭代逐步建立，禁止提前批量生成空洞文档。
- 每完成一个版本里程碑，重新评估范围、风险和下一阶段优先级。
- 如果实际业务证据与本蓝图冲突，以真实业务问题为准，并记录调整原因。
