# I1.1 仓库主数据方案

- 状态：后端实现完成，仓库级验收通过
- 所属迭代：I1 仓储主数据
- 权威实现模块：`backend/mdop-master-data`
- 最后更新：2026-07-21

## 1. 目标

I1.1 以单工厂、中型电子装配企业为参考场景，建立仓库主数据的最小闭环，为后续库区、库位、物料策略、库存、入库和出库提供稳定引用。

本切片只解决仓库本身的定义、分类、查询、启停和受控修改，不提前实现库存业务。

## 2. 业务边界

### 2.1 本期包含

- 新增仓库。
- 查询仓库详情。
- 按关键词、用途、形态、管理类别和状态分页查询。
- 修改仓库名称和备注。
- 仓库停用后修改结构属性。
- 启用和停用仓库。
- 乐观版本控制。
- 创建和修改审计。
- 软删除字段预留及查询隔离。
- 统一业务和参数错误响应。

### 2.2 本期不包含

- 删除仓库的外部接口。
- 库区、库位及库位容量。
- 物料、原料和半成品主数据。
- 库存余额和库存流水。
- 入库、出库、移库、盘点和调拨。
- 仓库数据权限。
- 完整 IAM（身份与访问管理）和 JWT（JSON Web Token，令牌机制）。
- 消息、缓存和 WMS（仓库管理系统）业务表。
- 前端仓库管理页面。
- 未经业务确认的特殊储藏能力字段。

仓库主数据的生命周期只包含启用和停用。入库和出库属于后续库存单据及库存流水，不作为仓库主数据状态，也不自动改变仓库启停状态。

## 3. 多维仓库分类

仓库使用多个独立维度表达业务含义，不使用一个单一类型字段混合所有属性。

### 3.1 主要用途

| 枚举值 | 含义 |
|---|---|
| `RAW_MATERIAL` | 原材料仓 |
| `SEMI_FINISHED` | 半成品仓 |
| `FINISHED_GOODS` | 成品仓 |
| `LINE_SIDE` | 线边仓 |
| `MRO` | 维护、维修和运营物料仓 |

### 3.2 仓库形态

| 枚举值 | 含义 |
|---|---|
| `PHYSICAL` | 具有实际储存空间的物理仓库 |
| `LOGICAL` | 用于业务隔离或账务管理的逻辑仓库 |

### 3.3 管理类别

| 枚举值 | 含义 |
|---|---|
| `GENERAL` | 常规物料管理 |
| `HAZARDOUS_CHEMICAL` | 危险化学品管理 |

电子制造业中的恒温恒湿、防潮、防静电和特殊洁净要求属于储藏能力维度。本期尚未确认能力字典和组合规则，因此不增加相应字段，也不把这些能力混入管理类别。

实际仓库由多个维度组合表达，例如：

- 常规原材料物理仓：`RAW_MATERIAL + PHYSICAL + GENERAL`
- 危险化学品原材料仓：`RAW_MATERIAL + PHYSICAL + HAZARDOUS_CHEMICAL`
- 逻辑半成品仓：`SEMI_FINISHED + LOGICAL + GENERAL`

## 4. 字段契约

| 字段 | 类型及约束 | 业务规则 |
|---|---|---|
| `id` | `BIGINT`，数据库自增 | 仓库唯一标识，创建后不可修改 |
| `code` | 最长 32 字符，唯一，非空 | 只允许大写字母、数字和连字符；创建后不可修改 |
| `name` | 最长 100 字符，非空 | 启用或停用状态均可修改 |
| `purpose` | 仓库用途枚举，非空 | 只有停用后才能修改 |
| `form` | 仓库形态枚举，非空 | 只有停用后才能修改 |
| `management_category` | 管理类别枚举，非空 | 只有停用后才能修改 |
| `status` | `ENABLED` 或 `DISABLED` | 新增时默认启用 |
| `remark` | 最长 500 字符，可空 | 启用或停用状态均可修改 |
| `version` | 非负整数 | 用于乐观并发控制 |
| `created_by` | 最长 64 字符，非空 | 创建人 |
| `created_at` | UTC 时间，非空 | 创建时间 |
| `updated_by` | 最长 64 字符，非空 | 最后修改人 |
| `updated_at` | UTC 时间，非空 | 最后修改时间 |
| `deleted_by` | 最长 64 字符，可空 | 软删除人，当前无删除接口 |
| `deleted_at` | UTC 时间，可空 | 软删除时间，当前无删除接口 |

仓库编码格式为：

```text
^[A-Z][A-Z0-9-]{1,31}$
```

数据库唯一约束不排除软删除记录，因此仓库编码在整个历史范围内永久唯一，不允许删除后重新使用。

## 5. 核心业务规则

1. 新增仓库默认状态为 `ENABLED`。
2. 仓库编码和 ID 创建后不可修改。
3. 启用状态下只能修改名称和备注。
4. 修改用途、形态或管理类别前必须先停用仓库。
5. 状态变更和修改请求必须携带当前版本号。
6. 版本号不一致时拒绝写入，不允许覆盖其他操作产生的修改。
7. 业务规则失败时整个事务回滚，不允许出现部分字段更新。
8. 软删除仓库不出现在列表查询中，详情查询按不存在处理。
9. 当前不提供 `DELETE` 接口。
10. 所有写操作记录当前认证用户名和 UTC 时间。

## 6. 数据库实现

权威迁移脚本：

`backend/mdop-master-data/src/main/resources/db/migration/masterdata/V202607210001__create_mdm_warehouse.sql`

权威表名：

`mdm_warehouse`

数据库负责保证：

- 主键约束。
- 仓库编码唯一约束。
- 用途、形态、管理类别和状态枚举检查。
- 版本号非负。
- `deleted_by` 和 `deleted_at` 必须同时为空或同时存在。
- UTF-8 四字节字符集。

已经在任何环境执行过的迁移脚本禁止修改。后续数据库变化只能新增 Flyway（数据库版本迁移工具）迁移。

## 7. API 契约

API（应用程序编程接口）统一前缀：

`/api/master-data/warehouses`

| 方法 | 路径 | 用途 |
|---|---|---|
| `POST` | `/api/master-data/warehouses` | 新增仓库 |
| `GET` | `/api/master-data/warehouses/{id}` | 查询仓库详情 |
| `GET` | `/api/master-data/warehouses` | 分页查询仓库 |
| `PUT` | `/api/master-data/warehouses/{id}` | 修改仓库 |
| `PUT` | `/api/master-data/warehouses/{id}/status` | 启用或停用仓库 |

不提供 `DELETE` 接口。

### 7.1 分页查询参数

| 参数 | 必填 | 说明 |
|---|---|---|
| `keyword` | 否 | 对仓库编码和名称进行包含查询； `%` 和 `_` 按普通字符处理 |
| `purpose` | 否 | 按主要用途筛选 |
| `form` | 否 | 按仓库形态筛选 |
| `managementCategory` | 否 | 按管理类别筛选 |
| `status` | 否 | 按启用状态筛选 |
| `page` | 否 | 从 1 开始，默认 1 |
| `size` | 否 | 默认 20，范围 1 至 100 |

列表按仓库编码升序返回。

### 7.2 响应字段

仓库响应包含：

- `id`
- `code`
- `name`
- `purpose`
- `form`
- `managementCategory`
- `status`
- `remark`
- `version`
- `createdBy`
- `createdAt`
- `updatedBy`
- `updatedAt`

分页响应额外包含：

- `items`
- `page`
- `size`
- `totalElements`
- `totalPages`

## 8. 错误契约

仓库业务错误和参数错误使用 RFC 9457 标准问题详情格式。

响应字段包括：

- `type`
- `title`
- `status`
- `detail`
- `instance`
- `code`
- `fieldErrors`，仅字段校验失败时提供

| 错误编码 | HTTP 状态 | 场景 |
|---|---:|---|
| `VALIDATION_FAILED` | 400 | 字段、分页、枚举或请求体格式不合法 |
| `WAREHOUSE_NOT_FOUND` | 404 | 仓库不存在或已经软删除 |
| `WAREHOUSE_CODE_DUPLICATE` | 409 | 仓库编码重复 |
| `WAREHOUSE_STRUCTURE_CHANGE_REQUIRES_DISABLED` | 409 | 启用状态下修改结构属性 |
| `WAREHOUSE_VERSION_CONFLICT` | 409 | 客户端版本号过期或数据库乐观锁冲突 |

安全过滤器在进入控制器前产生的未认证或无权限响应仍属于工程安全基线。本切片不提前扩展完整 IAM、JWT 或权限错误契约。

## 9. 安全与审计

- 所有仓库接口要求已认证用户。
- 写请求继续使用 CSRF（跨站请求伪造防护）基线。
- 创建人和修改人从 Spring Security 当前认证上下文读取。
- 应用不接受客户端传入审计人。
- 时间由统一 UTC `Clock` 提供。
- 本期不实现角色、菜单、仓库数据权限或完整登录流程。

## 10. 验收标准

### 10.1 数据库

- 空数据库能够执行唯一仓库迁移。
- Flyway 历史记录包含版本 `202607210001`。
- Hibernate 能够以 `ddl-auto=validate` 完成表结构校验。
- 仓库编码唯一约束和枚举检查有效。

### 10.2 正常路径

- 能新增仓库并返回 `201 Created` 和资源位置。
- 新增仓库默认启用。
- 能查询详情和分页列表。
- 关键词、用途、形态、管理类别和状态筛选有效。
- 软删除记录不会被查询到。
- 能修改名称和备注且编码保持不变。
- 停用后能够修改结构属性。
- 写操作正确记录认证用户名。
- 版本号随成功写操作递增。

### 10.3 失败路径

- 无 CSRF 令牌的写请求被拒绝。
- 非法字段、枚举、请求体和分页参数返回 400。
- 不存在和软删除仓库返回 404。
- 重复编码返回 409。
- 启用状态修改结构返回 409，数据库不发生部分更新。
- 旧版本写入返回 409，数据库内容保持不变。

### 10.4 回归要求

- Testcontainers（测试容器）必须使用随机宿主机端口和运行时临时凭据。
- 测试不得依赖 `.env.local` 或本机固定容器。
- 测试结束后不得遗留受管容器。
- 仓库级 `verify` 必须通过。
- 工作区变更必须与 I1.1 范围一致。
