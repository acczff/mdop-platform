# I0工程基线实施方案

> 所属项目：制造业数字化运营平台（MDOP）
>
> 迭代编号：I0
>
> 文档版本：V1.6
>
> 当前状态：方案已确认，实施中（Flyway、三类本地基础设施编排、共享 Testcontainers 测试基座、Spring Security 最小安全基线、前端 pnpm 工作区和 admin 应用基线，以及后端、前端格式化与质量门禁已完成）
>
> 最近更新：2026-07-17

## 1. I0目标

I0不开发具体WMS业务，目标是建立以后所有业务模块共同依赖的正式工程基线。

完成后必须达到：

- 新项目具有独立、有效、可回退的Git历史。
- 后端模块可以统一编译、测试和启动。
- 前端工作区可以统一检查、测试和构建。
- MySQL、RabbitMQ和Redis能够通过Docker Compose启动。
- Flyway能够从空数据库建立结构并完成校验。
- Spring Security具有最小可运行安全基线。
- 模块边界和禁止依赖能够被自动化测试检查。
- Testcontainers能够运行真实MySQL、RabbitMQ和Redis集成测试。
- 密钥和环境配置不进入代码仓库。
- 新开发只进入`mdop-platform`，旧项目停止新增功能。

## 2. 新旧项目关系

### 2.1 旧项目

```text
<本地工作区>\springboot-demo
<本地工作区>\springboot-web
```

处理原则：

- 完整保留，只作为业务和问题参考。
- 不再增加新功能。
- 不复制旧实体、旧表结构和旧认证代码。
- 不迁移旧数据库演示数据。
- 有价值的业务规则必须先解释、确认和测试，再在新项目重新实现。

### 2.2 新项目

```text
<本地工作区>\mdop-platform
├─ backend
├─ frontend
├─ deploy
├─ scripts
└─ README.md
```

新项目从第一次提交开始就是正式开发主线。

## 3. 后端结构

```text
backend
├─ pom.xml
├─ mvnw
├─ mvnw.cmd
├─ .mvn
├─ mdop-boot
├─ mdop-common
├─ mdop-security
├─ mdop-system
├─ mdop-master-data
├─ mdop-wms
├─ mdop-integration
└─ mdop-test-support
```

### 3.1 模块职责

| 模块 | 职责 |
|---|---|
| `mdop-boot` | 唯一应用入口、配置装配、Flyway统一执行 |
| `mdop-common` | 通用结果、基础异常、时间、ID等少量技术能力 |
| `mdop-security` | Spring Security、认证、授权和当前用户 |
| `mdop-system` | 用户、组织、角色、权限、菜单和操作审计 |
| `mdop-master-data` | 物料、供应商、仓库、库区和库位 |
| `mdop-wms` | 到货、收货、库存、出库和盘点等WMS领域 |
| `mdop-integration` | RabbitMQ、Inbox、Outbox、ERP/QMS适配和补偿任务 |
| `mdop-test-support` | Testcontainers、测试数据构造和共享测试工具 |

### 3.2 依赖原则

```text
boot
→ 负责装配所有模块

wms
→ common
→ security公开接口
→ master-data公开接口

integration
→ wms公开应用接口和领域事件
```

禁止：

- 业务模块依赖`mdop-boot`。
- 跨模块访问Repository。
- 将业务逻辑放入`mdop-common`。
- ERP、QMS消息DTO进入领域模型。
- Controller直接编写事务和库存规则。

### 3.3 WMS内部边界

```text
mdop-wms
├─ inbound
├─ inventory
├─ outbound
└─ stocktaking
```

每个业务边界按需采用：

```text
controller
application
domain
infrastructure
```

不为没有复杂规则的代码机械制造空层级。

## 4. 前端结构

前端使用pnpm工作区：

```text
frontend
├─ apps
│  └─ admin
├─ packages
│  └─ 按已确认职责创建共享包
├─ package.json
├─ pnpm-lock.yaml
└─ pnpm-workspace.yaml
```

I0 只在 `pnpm-workspace.yaml` 中预留 `packages/*`。`api-client`、`auth`、`ui`、`shared` 和 `config` 等共享包在职责明确后创建，不为满足目录外观提前建立空包。

未来允许增加：

```text
apps/warehouse-handheld
apps/workshop
apps/dashboard
```

前端原则：

- 从零建立Vue 3＋TypeScript管理端。
- TypeScript开启严格模式。
- API类型与页面状态分离。
- 登录、权限和数据范围进入`auth`包。
- 跨应用组件进入`ui`，业务页面组件留在所属应用。
- 不直接复制旧前端页面和Store。

## 5. 技术基线

| 组件 | 正式基线 |
|---|---|
| Java | Java 25 LTS |
| Spring Boot | 4.1.x最新补丁 |
| Maven | Maven Wrapper 3.9.x |
| Node.js | Node.js 24 LTS，最低 24.18.0且低于25 |
| pnpm | 11.13.0，由Corepack管理并通过`packageManager`固定 |
| MySQL | 8.4 LTS最新补丁 |
| RabbitMQ | 4.3.x最新补丁 |
| Redis | 8.2.x最新安全补丁 |
| 数据迁移 | Flyway |
| 后端测试与质量 | JUnit 5、Testcontainers、Maven Enforcer模块边界检查、JaCoCo |
| 前端 | Vue 3、TypeScript、Pinia、Vue Router、Vite |
| 前端测试 | Vitest，后期加入Playwright |
| 本地编排 | Docker Compose |

版本原则：

- 依赖版本优先由Spring Boot和父POM统一管理。
- 子模块不得随意覆盖公共依赖版本。
- Docker镜像固定具体补丁版本，不使用`latest`。
- 主版本升级必须建立ADR并完成回归测试。

## 6. 数据库与Flyway治理

### 6.1 新数据库

```text
旧库：springboot_demo，只读参考
新库：mdop，从空库建立
```

不复制旧表和旧演示数据。需要的数据在业务确认后通过正式迁移或开发种子数据重新建立。

### 6.2 迁移归属

```text
mdop-system/src/main/resources/db/migration/system
mdop-master-data/src/main/resources/db/migration/masterdata
mdop-wms/src/main/resources/db/migration/wms
mdop-integration/src/main/resources/db/migration/integration
```

只有`mdop-boot`启动Flyway并统一加载所有位置。

### 6.3 命名规则

```text
V202607110001__system_init.sql
V202607110002__master_data_init.sql
V202607110003__wms_inbound_init.sql
```

- 版本号全局唯一。
- 已发布迁移不得修改。
- 结构变化只能新增迁移。
- Hibernate固定`ddl-auto: validate`。
- 禁止保留平行的`init.sql`建库方式。
- 正式数据和开发演示数据分开。
- Flyway校验失败时应用拒绝启动。
- Testcontainers从空库执行全部迁移。

## 7. RabbitMQ正式定位

- ERP、WMS、QMS跨系统业务采用正式事件集成。
- WMS通过Inbox实现消费去重。
- WMS通过事务Outbox实现可靠发布。
- 投递语义为至少一次，业务结果通过双重幂等保证只生效一次。
- 具备重试、死信、人工重放、定时补偿和对账能力。
- RabbitMQ不是数据库事务的组成部分。
- Redis Pub/Sub不能替代正式业务事件。

## 8. Redis正式定位

### 8.1 第一批用途

- 登录失败次数和短期锁定
- 接口限流
- Token撤销快速查询
- 用户权限和菜单缓存
- 字典、物料等主数据缓存
- 短期防重复操作快速拦截
- 热点查询加速

### 8.2 禁止用途

- 不作为库存余额唯一数据源。
- 不仅依赖Redis锁保证库存正确性。
- 不作为正式幂等结果的唯一存储。
- Inbox和Outbox不能只存Redis。
- 不使用Redis Pub/Sub替代RabbitMQ。
- Redis故障后权限不能默认放行。

### 8.3 使用规范

每个Redis功能必须定义：

- Key命名空间
- 数据结构
- TTL
- 写入与失效策略
- 穿透处理
- 故障降级方式
- 可接受不一致时间
- 容量和监控指标

最终职责：

```text
MySQL：业务事实和最终数据源
RabbitMQ：可靠跨系统事件
Redis：缓存、限流和短期状态
```

## 9. 环境与部署

### 9.1 环境

```text
local   本地开发
test    Testcontainers自动化测试
dev     集成开发环境
prod    正式环境
```

### 9.2 部署目录

```text
deploy
├─ compose
│  ├─ compose.local.yml
│  └─ compose.dev.yml
├─ docker
│  ├─ backend.Dockerfile
│  └─ frontend.Dockerfile
├─ monitoring
└─ env
   └─ .env.example
```

### 9.3 配置和容器规则

- 仓库只提交安全默认值和`.env.example`。
- 密码、JWT密钥和RabbitMQ凭据不提交。
- 环境差异由环境变量提供。
- 所有环境使用同一套结构迁移。
- 不通过修改源码切换环境。
- 多阶段构建，应用以非root用户运行。
- MySQL、RabbitMQ、Redis和应用都配置健康检查。
- 启动依赖健康状态，不只依赖容器进程已启动。
- 数据卷、日志和备份目录明确。

## 10. Git与质量门禁

### 10.1 独立仓库

```text
<本地工作区>\mdop-platform\.git
```

- `main`始终保持可构建、可测试。
- 使用短生命周期功能分支。
- 暂不建立长期`develop`分支。
- 后续接入远程仓库后通过PR和CI合并。

### 10.2 后端门禁

- Maven Enforcer
- 依赖收敛检查
- Spotless格式检查
- ArchUnit或模块边界测试
- JUnit 5＋Testcontainers
- JaCoCo覆盖率报告
- Flyway空库迁移测试

### 10.3 前端门禁

- TypeScript严格模式
- ESLint
- Prettier
- 类型检查
- Vitest
- 生产构建检查
- 后期加入Playwright

### 10.4 仓库基础文件

```text
.gitignore
.editorconfig
.gitattributes
README.md
CONTRIBUTING.md
```

## 14. 变更规则

- 实施发现版本、插件或运行环境不兼容时，先记录证据，再调整基线。
- 任何模块新增依赖必须说明原因和影响。
- 任何跨模块访问必须经过公开接口或事件。
- 重大技术选择变更建立ADR。
- I0结束前不得提前开发WMS业务以绕过工程问题。
